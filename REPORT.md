## 1. Реализация Redis-кэша
_TODO: Опишите, что именно кэшируется и с какими ключами._

Нужно указать:
- ключ каталога;
- ключ карточки заказа;
- TTL;
- логику cache hit/miss.

Ключи кэша:
Каталог товаров: catalog:v1 — единый ключ для всего списка товаров. Версионирование (v1) позволяет инвалидировать весь каталог одновременно при необходимости.
Карточка заказа: order_card:v1:{order_id} — отдельный ключ для каждого заказа, где {order_id} — UUID конкретного заказа.

TTL :
Каталог: 300 секунд (5 минут) — каталог обновляется редко, долгий TTL снижает нагрузку на БД.
Карточка заказа: 60 секунд — статус заказа меняется чаще, поэтому TTL короче.

Когда приходит запрос с use_cache=true, сервер сначала смотрит в Redis:
Ключ найден (HIT) — данные сразу отдаются клиенту. PostgreSQL не трогается. 
Ключ не найден (MISS) — сервер идёт в PostgreSQL, берёт данные, сохраняет их в Redis с TTL 300 секунд и отвечает клиенту. Следующие запросы уже попадут в HIT.
При use_cache=false Redis игнорируется полностью — каждый раз идём в PostgreSQL. Этот режим используется в бенчмарках как точка сравнения.

## 2. Демонстрация неконсистентности (намеренно сломанный сценарий)
_TODO: Опишите шаги и результаты stale cache._

Минимум:
1. Прогрев кэша.
2. Изменение заказа в БД без инвалидации.
3. Повторный запрос из кэша.
4. Чем stale ответ отличается от данных в БД.

Шаги:
Прогрев кэша. Выполняется запрос GET /cache-demo/order/{order_id}. Сервер обращается к PostgreSQL, получает данные (status: created) и сохраняет их в Redis по ключу order_card:v1:{order_id}.
Изменение заказа в БД без инвалидации. Через эндпоинт POST /cache-demo/mutate-without-invalidation статус заказа в таблице orders обновляется до cancelled. Операция DEL order_card:v1:{order_id} в Redis не выполняется.
Повторный запрос из кэша. Клиент снова делает GET /cache-demo/order/{order_id}. Сервер находит ключ в Redis (Cache HIT) и возвращает старые данные.
Разница между stale-ответом и реальными данными:
| Источник        | status    |
| --------------- | --------- |
| Redis (кэш)     | created   |
| PostgreSQL (БД) | cancelled |

Пользователь видит заказ в статусе «создан», хотя в действительности он уже отменён — это и есть stale data. В финансовых системах подобная ситуация может приводить к попыткам повторной оплаты уже отменённого заказа.

## 3. Починка через событийную инвалидацию
_TODO: Опишите механизм события и инвалидации._

Нужно указать:
- где генерируется событие изменения заказа;
- где обрабатывается событие;
- какие ключи инвалидируются и почему.

Механизм: при изменении заказа публикуется событие, которое перехватывается обработчиком, удаляющим устаревшие ключи из Redis.
Где генерируется событие:
В файле cache_demo_routes.py, эндпоинт POST /cache-demo/mutate-with-event-invalidation. После выполнения UPDATE orders SET status = ... вызывается функция из cache_events.py:
python

    await publish_order_changed_event(order_id=order_id)

Где обрабатывается событие:
В файле cache_events.py. Обработчик подписан на Redis-канал cache_invalidation через механизм Pub/Sub. При получении сообщения выполняет удаление соответствующих ключей:
python

    await redis.delete(f"order_card:v1:{order_id}")
    await redis.delete("catalog:v1")
    
Какие ключи инвалидируются и почему:
order_card:v1:{order_id} — напрямую связан с изменённым заказом, содержит устаревший статус.
catalog:v1 — инвалидируется опционально, если агрегат каталога включает статусы заказов.
TTL служит страховочным механизмом: даже если событие инвалидации по каким-то причинам не дошло (сбой сети, рестарт контейнера), кэш автоматически устареет через 60 секунд и при следующем запросе данные будут актуализированы из PostgreSQL.


## 4. Rate limiting endpoint оплаты через Redis
_TODO: Опишите реализацию лимитов._

Нужно указать:
- policy (например N запросов за M секунд);
- ключ лимита (по user_id/ip);
- поведение при превышении (`429`);
- какие заголовки возвращаются клиенту.

Реализован лимит 10 запросов за 60 секунд на эндпоинт POST /api/payments/retry-demo.
Ключ привязан к IP-адресу клиента: rate_limit:{client_ip}:payments. IP выбран вместо user_id, чтобы защита работала даже для неаутентифицированных запросов.
Алгоритм — Sliding Window через Redis Sorted Set: при каждом запросе добавляется метка времени, старые метки (старше 60 секунд) удаляются, и если оставшихся меток больше 10 — запрос отклоняется.
При превышении лимита сервер возвращает 429 Too Many Requests с заголовками:

    text
    X-RateLimit-Limit: 10
    X-RateLimit-Remaining: 0
    X-RateLimit-Reset: 42
    Retry-After: 42
    
где Retry-After — количество секунд, через которое лимит сбросится и клиент снова сможет делать запросы.

## 6. Выводы
_TODO: Сформулируйте 3–5 практических выводов._

Рекомендуется осветить:
1. Когда кэш даёт выигрыш, а когда нет.
2. Почему инвалидация сложнее, чем кэширование.
3. Почему rate limiting полезен даже при наличии бизнес-валидаций.


1. Когда кэш даёт выигрыш, а когда нет.
Кэш эффективен для данных с высокой частотой чтения и низкой частотой изменений: каталоги товаров, справочники, агрегированная статистика. Кэш вреден для данных, где нельзя допустить устаревания даже на секунду: текущий баланс счёта при списании, статус активной транзакции. В таких случаях кэш не только бесполезен, но и опасен.
2. Почему инвалидация сложнее, чем само кэширование.
Добавить SET key value EX 300 в Redis — задача на одну строку. Но гарантировать, что кэш будет сброшен при каждом изменении данных — это требует анализа всех точек записи в системе, атомарности операций «записать в БД + инвалидировать кэш», а также устойчивости к частичным сбоям (что если контейнер упал между записью и инвалидацией?). Именно поэтому TTL является обязательным дополнением к событийной инвалидации — он страхует от любых «дыр».
3. Почему rate limiting полезен даже при наличии бизнес-валидаций.
Бизнес-валидации (например, проверка идемпотентности из ЛР4) защищают от логических дублей. Rate limiting защищает от физической перегрузки сервера. Даже если все 10 000 запросов в секунду валидны логически, они могут положить базу данных — и тогда перестанут работать все пользователи. Rate limiting — это первый барьер защиты, который срабатывает ещё до того, как запрос дошёл до бизнес-логики.
4. Redis как универсальный инструмент.
В рамках одной лабораторной работы Redis выполнял три совершенно разные роли: кэш (хранение результатов SELECT), брокер событий (Pub/Sub для инвалидации) и счётчик запросов (Sorted Set для rate limiting). Это демонстрирует, что Redis — это не просто «быстрый кэш», а полноценный инструмент для построения высоконагруженных систем.
5. Измеримость результата.
Внедрение кэша дало прирост RPS более чем в 25 раз при снижении латентности на 98%. Нагрузка на PostgreSQL упала практически до нуля при прогретом кэше. Это подтверждает, что кэширование — одна из наиболее эффективных оптимизаций для read-heavy систем, и её эффект поддаётся точному измерению через нагрузочное тестирование.
